# OutlinerGod — REST API Contract

**Backend**: Fastify v5 · Drizzle + better-sqlite3 · SQLite
**Auth**: Google ID token → backend JWT (jose v6)
**Base URL**: all endpoints prefixed `/api` except `/health`

---

## Common Definitions

### JWT
Short-lived access token (1 h). All protected endpoints require:
```
Authorization: Bearer <jwt>
```

### HLC Timestamp (string)
Hybrid Logical Clock string — lexicographically sortable, used for per-field LWW merge.
Format: `<wall_ms_16hex>-<counter_4hex>-<device_id>`
Example: `0000017b05a3a1be-0000-a1b2c3d4`
- `wall_ms_16hex` — 16-char hex of Unix ms timestamp
- `counter_4hex` — 4-char hex logical counter (incremented on clock ties)
- `device_id` — originating device ID (UUID, stable per installation)

### Fractional Index (string)
TEXT column, lexicographic sort determines sibling order. Generated by `fractional-indexing-jittered`.
Examples: `"V"`, `"a0"`, `"a1V"`. Always `ORDER BY sort_order ASC, id ASC` to break ties.

### Soft Delete
`deleted_at` is `null` (active) or a Unix ms timestamp (deleted). Tombstones retained 90 days before purge.

### File Ownership
Every uploaded file has a row in the `files` metadata table associating it with a `user_id`. This enables ownership checks on GET and DELETE operations. The `files` table has columns: `filename` (TEXT PK, `<uuid>.<ext>`), `user_id` (TEXT NOT NULL), `node_id` (TEXT NULLABLE), `mime_type` (TEXT), `size` (INTEGER), `created_at` (INTEGER).

## 4. Sync

Sync uses **per-field Hybrid Logical Clock (HLC) + Last-Write-Wins (LWW)**. Every syncable field has a companion `_hlc` column. The merge rule: for each field independently, the version with the lexicographically higher HLC wins. Deletion wins over concurrent edits — if `deleted_hlc` > all other field HLCs, the node is deleted everywhere.

**Sync cycle order**: always PULL → then PUSH to minimize conflicts.

### GET /api/sync/changes

**Auth required**: Bearer JWT
**Purpose**: Pull all changes (nodes, documents, settings, bookmarks) modified after `since` HLC. Returns tombstones (deleted records) so clients can apply deletions. Client uses `server_hlc` from the response as the `since` value for the next pull.

**Query parameters**

| Parameter | Type | Required | Description |
|-----------|------|----------|-------------|
| `since` | string (HLC) | ✓ | Pull all records with any field HLC > this value. Use `"0"` as the canonical sentinel on first sync. This works because all valid HLC strings begin with a 16-char hex wall clock (`0000017b...`), which is always lexicographically greater than `"0"`. |
| `device_id` | string | ✓ | Requesting device ID; server excludes changes that originated from this device to avoid echo |

**Response 200**

| Field | Type | Description |
|-------|------|-------------|
| `server_hlc` | string (HLC) | Server's current HLC at time of response. Store as `last_sync_hlc`. |
| `nodes` | NodeSyncRecord[] | Changed node records (includes tombstones where `deleted_at` is set) |
| `documents` | DocumentSyncRecord[] | Changed document/folder records |
| `settings` | SettingsSyncRecord \| null | Changed settings (null if none since `since`) |
| `bookmarks` | BookmarkSyncRecord[] | Changed bookmark records |

**Errors**

| Status | Reason |
|--------|--------|
| 400 | Missing or malformed `since` or `device_id` |
| 401 | Unauthorized |

---

### POST /api/sync/changes

**Auth required**: Bearer JWT
**Purpose**: Push locally pending changes to the server. Server applies per-field LWW merge using HLC comparison. For each field, if the incoming HLC > stored HLC, the incoming value wins. If the incoming HLC ≤ stored HLC, the stored value wins and the server returns the winning version in `conflicts`. Deletion (non-null `deleted_at` with highest `deleted_hlc`) always wins over concurrent field edits.

**Request body**

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `device_id` | string | ✓ | Originating device ID |
| `nodes` | NodeSyncRecord[] | ✗ | Pending node changes (omit if none) |
| `documents` | DocumentSyncRecord[] | ✗ | Pending document changes |
| `settings` | SettingsSyncRecord | ✗ | Pending settings changes |
| `bookmarks` | BookmarkSyncRecord[] | ✗ | Pending bookmark changes |

**Response 200**

| Field | Type | Description |
|-------|------|-------------|
| `server_hlc` | string (HLC) | Server's HLC after processing. Update `last_sync_hlc`. |
| `accepted_node_ids` | string[] | IDs of nodes where all incoming fields won LWW merge |
| `accepted_document_ids` | string[] | IDs of documents where all incoming fields won |
| `accepted_bookmark_ids` | string[] | IDs of bookmarks where all incoming fields won |
| `conflicts` | ConflictsObject | Records where server's stored version won on at least one field |

**ConflictsObject**

| Field | Type | Description |
|-------|------|-------------|
| `nodes` | NodeSyncRecord[] | Full node sync records for each node with at least one conflicting field. Client applies these locally via LWW (server version will always win since its HLC is already higher). |
| `documents` | DocumentSyncRecord[] | Same pattern for documents |
| `bookmarks` | BookmarkSyncRecord[] | Same pattern for bookmarks |
| `settings` | SettingsSyncRecord \| null | Settings winning version if any field conflicted |

**Errors**

| Status | Reason |
|--------|--------|
| 400 | Missing `device_id`, malformed HLC strings, or unknown record IDs |
| 401 | Unauthorized |
| 413 | Payload too large (max 5 MB) |

## 5. Attachments

File attachments are stored at `/data/uploads/<uuid>.<ext>` in the Docker volume. The server generates and validates UUID-based filenames. Path traversal is prevented by validating filename against `^[0-9a-f-]{36}\.[a-z0-9]+$` before any file operation.

### POST /api/files

**Auth required**: Bearer JWT
**Purpose**: Upload a file attachment. Client must compress images to < 1 MB before uploading. Server associates the file with the authenticated user.

> **Size limit**: Default server maximum is 50 MB per file, configurable via the `MAX_UPLOAD_BYTES` environment variable. Images must be compressed to < 1 MB by the Android client before upload.

**Request**: `multipart/form-data`

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `file` | binary | ✓ | File content |
| `node_id` | string | ✗ | Node UUID to associate the attachment with (informational; server does not enforce referential integrity) |

**Response 201**

| Field | Type | Description |
|-------|------|-------------|
| `url` | string | Canonical URL: `/api/files/<uuid>.<ext>` |
| `uuid` | string | Generated UUID (without extension) |
| `filename` | string | Full filename: `<uuid>.<ext>` |
| `size` | number | Stored file size in bytes |
| `mime_type` | string | Detected MIME type |

**Errors**

| Status | Reason |
|--------|--------|
| 400 | Missing file part or malformed multipart body |
| 401 | Unauthorized |
| 413 | File exceeds server-configured max size |

### PUT /api/settings

### POST /api/bookmarks

**Purpose**: Create a new bookmark.

### PATCH /api/bookmarks/:id

**Purpose**: Update a bookmark's title or sort_order. All fields optional; only supplied fields are updated.

## 8. Export

### GET /api/export

**Auth required**: Bearer JWT
**Purpose**: Generate and stream a ZIP file of the authenticated user's entire data — all documents and folders (as JSON), all nodes (as JSON per document), and all file attachments from the Docker volume.

**Query parameters**: None

**Response 200**: `application/zip` binary stream. Header: `Content-Disposition: attachment; filename="outlinergod-export-<timestamp>.zip"`

ZIP contents:
- `documents.json` — all documents and folders
- `nodes/<document_uuid>.json` — one file per document containing all its nodes
- `attachments/<uuid>.<ext>` — all uploaded files

**Errors**

| Status | Reason |
|--------|--------|
| 401 | Unauthorized |

---

## 9. Health

### GET /health

**Auth required**: None
**Purpose**: Container health check. Used by Docker `HEALTHCHECK` directive and load balancers.

**Response 200**

| Field | Type | Description |
|-------|------|-------------|
| `status` | `"ok"` | Always `"ok"` if the process is running |
| `version` | string | Backend semver (e.g., `"1.0.0"`) |
| `db` | `"ok"` \| `"error"` | Result of a lightweight SQLite ping (`SELECT 1`) |
| `uptime` | number | Process uptime in seconds |

**Errors**: None — if the process is unreachable, Docker considers the container unhealthy.
