## Tech Debt (Non-Blocking)

### TD-1: SettingsSyncRecord.id = userId (cosmetic)
**Phase:** 08-settings-sync-push
`toSettingsSyncRecord()` sets `id = userId` (SyncMappers.kt:159). Functionally safe — backend upsert uses `@id` binding and the UUID userId is a valid unique key. Would become inconsistent only if `GET/PUT /api/settings` REST endpoints were directly wired from Android alongside the sync path.

### TD-2: CreateDocumentRequest.kt — orphaned dead code
**Phase:** 04-android-core / 05-sync-integration-fixes
`CreateDocumentRequest.kt` is never imported or called. The Android client creates documents in Room and syncs via `POST /api/sync/changes`, not `POST /api/documents` directly. Should be deleted or wired when direct document REST creation is needed (e.g., v0.5 import feature).

### TD-3: DocumentDetailScreen / BookmarksScreen — unreachable routes
**Phase:** 04-android-core
Routes registered in `AppNavHost.kt` but no navigation leads to them. Placeholders for v0.5 search-export-bookmarks phase. Harmless; low cleanup priority.

### TD-4: NodeDao.getPendingChanges lacks userId scope
**Phase:** 03-android-setup
`NodeDao.getPendingChanges` filters by `device_id` only. `DocumentDao` and `BookmarkDao` filter by both `user_id` and `device_id`. Safe in single-user self-hosted model; architecturally inconsistent. Low risk.

### TD-5: Phase 01 UAT never formally run
**Phase:** 01-backend-foundation
`01-UAT.md` has all tests as `[pending]`. All 5 acceptance criteria confirmed by transitive evidence (Phase 02 UAT + subsequent phases). No code risk; audit trail gap documented in `01-VERIFICATION.md`.

### Pre-existing: BookmarkDaoTest flaky
`BookmarkDaoTest.observeAllActive_excludesOtherUsers` — Robolectric/Room race. Passes on retry and in isolation. Pre-dates Phase 04. Not introduced by any v0.4 phase.
