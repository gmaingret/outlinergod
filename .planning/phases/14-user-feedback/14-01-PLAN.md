# Plan 14-01: Delete Undo via Toolbar Button

## Context
User feedback: "delete should be recover with the undo button in the toolbar."

Currently `deleteNode()` calls `nodeDao.softDeleteNodes(idsToDelete, ...)` but does NOT push to `undoStack`.
The toolbar Undo button (`canUndo`) only covers structural ops (indent/outdent/move/reorder).
`restoreNode()` only un-deletes a single node — subtree descendants stay deleted.

## What to Build

### Task A: Push to undoStack in deleteNode()
File: `NodeEditorViewModel.kt`

In `deleteNode()`, before calling `nodeDao.softDeleteNodes`:
1. Push the current `flatNodes` snapshot to `undoStack`
2. Store the list of IDs being deleted alongside the snapshot so `undo()` knows what to restore
3. Set `canUndo = true` in the reduce

Implementation: add `private val undoDeletedIds = ArrayDeque<List<String>>()` to ViewModel.
In `deleteNode()`:
```kotlin
pushUndoSnapshot(flatNodes)            // existing helper
undoDeletedIds.addLast(idsToDelete)    // parallel stack
reduce { state.copy(canUndo = true) }
```

### Task B: Extend undo() to restore soft-deleted nodes
File: `NodeEditorViewModel.kt`

In `undo()`, after calling `persistReorderedNodes(current, snapshot)`:
1. Pop from `undoDeletedIds` (if non-empty)
2. For each deleted ID, call `nodeDao.restoreNodes(ids, hlc, now)` — a new DAO method
3. The Flow collector in `loadDocument()` will automatically add the restored nodes back to flatNodes

Edge case: if `undoStack` was pushed by a non-delete op, `undoDeletedIds` pop would be misaligned.
Fix: always push to `undoDeletedIds` (push empty list for non-delete ops).
Change `pushUndoSnapshot()` to also push `emptyList<String>()` to `undoDeletedIds`.

### Task C: Add NodeDao.restoreNodes()
File: `NodeDao.kt`

```kotlin
@Query("UPDATE nodes SET deleted_at = NULL, deleted_hlc = :hlc, updated_at = :updatedAt WHERE id IN (:nodeIds)")
suspend fun restoreNodes(nodeIds: List<String>, hlc: String, updatedAt: Long)
```

### Task D: Fix restoreNode() to restore subtree
File: `NodeEditorViewModel.kt`

`restoreNode(nodeId)` currently only restores one node. Fix to also restore all descendants
that have the same `deleted_at` timestamp (indicating they were deleted in the same batch).

```kotlin
fun restoreNode(nodeId: String) = intent {
    val node = nodeDao.getNodeById(nodeId).first() ?: return@intent
    val deletedAt = node.deletedAt ?: return@intent
    // Find all nodes in same document deleted at the same timestamp
    val allNodes = nodeDao.getNodesByDocumentIncludingDeleted(state.documentId)
    val idsToRestore = allNodes
        .filter { it.deletedAt == deletedAt }
        .map { it.id }
    ...
    nodeDao.restoreNodes(idsToRestore, hlc, now)
}
```

Add to `NodeDao`:
```kotlin
@Query("SELECT * FROM nodes WHERE document_id = :documentId")
suspend fun getNodesByDocumentIncludingDeleted(documentId: String): List<NodeEntity>
```

### Task E: Also push to undoStack from context menu Delete
The context menu calls `deleteNode()` which will now push to undoStack (Task A covers this).
The snackbar undo (swipe-to-delete) calls `restoreNode()` which will now restore subtrees (Task D).
No additional UI changes needed — toolbar Undo will also work for context menu Delete.

## Input
- `NodeEditorViewModel.kt` — `deleteNode()`, `undo()`, `pushUndoSnapshot()`, `restoreNode()`
- `NodeDao.kt` — add `restoreNodes()`, `getNodesByDocumentIncludingDeleted()`

## Output
- Toolbar Undo button restores deleted nodes (including subtree)
- Snackbar undo (swipe-to-delete) also restores subtree
- `undoStack` and `undoDeletedIds` stay in sync

## Acceptance Criteria
1. Delete a node (with children) via context menu → toolbar Undo button becomes active
2. Tap Undo → node AND all its children reappear
3. Swipe-to-delete a node with children → snackbar "Undo" → node AND all children reappear
4. Undo/Redo still works normally for structural ops (indent/outdent/move)
