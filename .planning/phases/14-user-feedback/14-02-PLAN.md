# Plan 14-02: Drag Reparenting Fix

## Context
User feedback: "drag a L1 node in the middle of L2, it becomes the new parent of the L2 nodes below.
Instead it should move at a L2 position with it's own children."

## Root Cause Analysis

In `reorderNodes()`, after inserting the block at `insertAt`, the code only enforces depth in
the "too deep" direction (block head depth > maxAllowedDepth). It does NOT handle the case
where the block head ends up at a SHALLOWER depth than its surroundings.

When an L1 node (depth=0) is dropped between two L2 nodes (depth=1):
- `nodeAbove` = an L2 node (depth=1)
- `maxAllowedDepth` = 2
- `blockHeadDepth` = 0
- 0 < 2 → the depth enforcement block is skipped entirely

Result: the L1 node keeps its original `parentId` (doc root) but appears between L2 siblings
in the flat list. `recomputeFlatNodes` assigns sort orders using existing parentIds, so:
- The L1 node's sort order is updated relative to other doc-root children (not the L2 siblings)
- Nothing meaningful is persisted
- On next DB load, the L1 node reverts to after its original L2 siblings

**Visual bug**: the optimistic flat list shows L1(depth=0) followed by L2_below(depth=1),
making L2_below look like a child of L1 visually (depth rendering: deeper = child).
This is the "becomes new parent" the user sees.

## What to Build

### Task A: Update block head's depth and parentId to match drop context
File: `NodeEditorViewModel.kt`, `reorderNodes()`

Replace the current "enforce depth if too deep" block with a general "adopt target context" block:

```kotlin
val nodeAbove = withoutBlock.getOrNull(insertAt - 1)
val targetDepth = nodeAbove?.depth ?: 0
val targetParentId = when {
    targetDepth == 0 -> state.documentId ?: state.rootNodeId
    else -> nodeAbove!!.entity.parentId  // become sibling of nodeAbove
}

val blockHeadDepth = withoutBlock[insertAt].depth
if (blockHeadDepth != targetDepth) {
    val depthDelta = blockHeadDepth - targetDepth  // positive = need to go shallower, negative = deeper
    val newParentId = if (blockHeadDepth > targetDepth) {
        // Too deep: reparent to nodeAbove's parent (existing behavior)
        nodeAbove?.entity?.id ?: (state.documentId)
    } else {
        // Too shallow: reparent to become sibling of nodeAbove
        targetParentId
    }
    for (i in insertAt until insertAt + blockSize) {
        val n = withoutBlock[i]
        withoutBlock[i] = n.copy(
            entity = if (i == insertAt) n.entity.copy(parentId = newParentId) else n.entity,
            depth = n.depth - depthDelta,
        )
    }
}
```

**Design decision**: vertical drag adopts the depth of the surrounding context. This matches the
user's expectation. The user explicitly wants reparenting to happen when dropping between nodes
of a different depth level.

**Edge case — rootNodeId (zoom mode)**: when `state.rootNodeId != null`, the minimum depth is 0
but the parentId for depth=0 should be `state.rootNodeId`, not `state.documentId`. Adjust:
```kotlin
val minParentId = state.rootNodeId ?: state.documentId
```

**Edge case — nodeBelow conflict**: if nodeBelow exists and is at a deeper depth than targetDepth+1,
it means the block is being inserted as the LAST CHILD of a node. In this case targetDepth+1 is
the appropriate depth. Rule: use min(targetDepth, nodeBelow's depth - 1) if nodeBelow is present.
Keep it simple for now — only use nodeAbove for depth determination.

### Task B: Fix recomputeFlatNodes to also update parentId from flat structure
File: `NodeEditorViewModel.kt`, companion object

Currently `recomputeFlatNodes` only assigns sort orders and relies on existing `parentId`.
After Task A, `parentId` of the block head IS updated before calling `recomputeFlatNodes`, so
`recomputeFlatNodes` will correctly group the reparented node with its new siblings.
No change to `recomputeFlatNodes` needed.

### Task C: Verify rootDocumentId is always available
In `reorderNodes`, `state.documentId` is always set (initialized from `loadDocument()`).
When zoomed in (`rootNodeId != null`), `state.documentId` still holds the real document ID.
No change needed.

## Input
- `NodeEditorViewModel.kt` — `reorderNodes()` depth enforcement block

## Output
- Dragging L1 into middle of L2 nodes: L1 becomes L2 (depth=1, parentId=L2s' parent)
- Dragging L2 up to root level: L2 becomes L1 (depth=0, parentId=documentId)
- Dragging within same depth level: sort order changes only, no parentId change
- Children move with parent and shift depth by the same delta

## Acceptance Criteria
1. Drag an L1 node (with children) between two L2 children of another node → L1 becomes L2,

3. Reorder two siblings within the same parent → positions swap, no reparenting
